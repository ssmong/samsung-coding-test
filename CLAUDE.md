# 워크스페이스 개요

삼성 SW 역량테스트 A형 대비를 위한 백준 문제풀이 저장소.
시험 정보·출제 경향은 [docs/samsung-coding-test.md](docs/samsung-coding-test.md) 참고.

- **언어:** Python 3 (단, `sys.stdin.readline`, `sys.setrecursionlimit` 등 최적화 필수)
- **난이도 표기:** solved.ac 기준. 같은 티어에서는 **숫자가 작을수록 어렵다** (실버1 > 실버5).
- **출처:** 주로 백준(BOJ).

# 디렉토리 규칙

```
problems/<카테고리>/<티어>/<백준번호>-<슬러그>/
├── problem.md      # 문제 정리 (Claude가 생성)
├── solution.py     # 사용자 풀이
└── notes.md        # (선택) 풀이 후 회고·실수 노트
```

- `<티어>`: `bronze` / `silver` / `gold` / `platinum`
- `<슬러그>`: 영문 소문자 + 하이픈 (예: `1260-dfs-and-bfs`)
- 한 문제가 여러 알고리즘에 걸치면 **가장 핵심 알고리즘** 폴더에 두고, 다른 카테고리에서는 `notes.md`에 교차 참조.

# Claude 사용 워크플로

## 1. 문제 등록

사용자가 백준 문제 본문(제목·난이도·시간/메모리·문제·입력·출력·예제)을 그대로 붙여넣는다.

Claude는 다음을 수행:
1. 카테고리·티어 판정 → 폴더 경로 결정 (애매하면 사용자에게 확인)
2. `problem.md` 생성 — 아래 템플릿 따른다
3. 빈 `solution.py` 스텁은 만들지 않는다 (사용자가 직접 작성)

### `problem.md` 템플릿

```markdown
---
source: BOJ
number: <번호>
title: <한글 제목>
tier: <티어>  # e.g. silver-2
url: https://www.acmicpc.net/problem/<번호>
algorithms: [<주 알고리즘>, <부 알고리즘>]
time_limit: <초>
memory_limit: <MB>
---

# <번호>. <제목> [<티어>]

## 문제
<원문 그대로>

## 입력
<원문 그대로>

## 출력
<원문 그대로>

## 예제
### 입력 1
\`\`\`
...
\`\`\`
### 출력 1
\`\`\`
...
\`\`\`
(예제가 여러 개면 반복)

## 메모
- 제약 조건 핵심: <N, M 범위, 시간 복잡도 추정>
- 주의할 엣지 케이스: <빈 입력, 자기 자신 등>
```

## 2. 풀이 도움 요청 (3단계 도움 시스템)

사용자가 풀이 중 막혔을 때 **반드시 도움 레벨을 명시**한다. Claude는 그 레벨을 넘지 않는다.

### L1 — 힌트만 (`/help L1` 또는 "L1로 도와줘")
- **코드 금지.** 자연어 힌트만.
- 어떤 알고리즘/자료구조 카테고리인지, 어떤 관점으로 접근해야 할지, 놓치기 쉬운 엣지 케이스 한 가지.
- 한 번에 한 가지 힌트만. 추가 요청 시 다음 힌트.

### L2 — 틀 잡아주기 (`L2`)
- 의사코드(pseudocode) 또는 함수 시그니처·골격만 제공.
- 핵심 자료구조 선언과 메인 루프 구조까지. **핵심 로직 본문은 비워둔다** (`# TODO`).
- 사용자가 채워넣을 부분을 명확히 표시.

### L3 — 완전 알려주기 (`L3`)
- 완성된 Python 풀이 + 복잡도 분석 + 왜 그 접근을 선택했는지.
- 실수하기 쉬운 부분을 코드 옆에 짧게 코멘트.
- L3 사용 후엔 사용자가 같은 알고리즘 다른 문제를 직접 풀어보도록 유도.

### 디버깅 모드 (`debug`)
- 사용자가 작성한 코드가 틀렸을 때 사용. 도움 레벨과 별개.
- 먼저 코드 읽고 **버그 위치만** 가리킨다 (실패 입력 + 한 줄 진단).
- 사용자가 추가 설명을 요청하면 그때 원인·수정 방향 제시.

## 3. 풀이 검토 (`review`)

이미 통과한 코드라도 더 빠르게/안전하게 만들 여지가 있다. `review` 요청 시 아래 항목을 **순서대로** 점검:

### A. 정확성 리스크 (먼저 본다)
- 입력 경계값 (N=1, M=0, 자기 자신, 음수, 0)
- 자료형 오버플로 (Python은 int 무한이라 드물지만, 곱셈으로 큰 값 → 시간 폭발 가능)
- 인덱스 0/1 혼용 실수
- BFS `visited` 표시 시점 (popleft가 아니라 enqueue 시점)
- 재귀 깊이 (`sys.setrecursionlimit` 누락)
- 동시 변경(simulation) 시 deepcopy/스냅샷 누락 → 같은 턴 내 객체끼리 영향 주는 버그
- 전역 변수와 함수 인자 혼용으로 인한 의도치 않은 상태 공유

### B. 시간 복잡도
- 큰 O 산정 → 입력 상한 대입 → 1초당 ~10^7 (Python) 기준으로 여유 확인
- 숨은 O(N) 연산: `list.pop(0)`, `in list`, `del list[i]`, 문자열 `+=` 누적
- 이중 정렬·중복 BFS 호출 등 redundant work

### C. Python 특화 최적화
- [Python 최적화 체크리스트](#python-최적화-체크리스트) 항목 전수 점검
- `for ... in zip(...)` vs 인덱스 루프
- `dict.get(k, default)` 패턴
- 핫 루프 안에서 attribute lookup 캐싱 (`append = q.append`)
- numpy는 백준에서 import 가능하지만 실전 시험에선 안 쓰는 게 안전

### D. 코드 명료성
- 매직 넘버 → 상수
- 깊은 중첩 → 헬퍼 함수 분리 (단, 함수 호출 오버헤드 vs 가독성 트레이드오프 명시)

### 출력 형식
1. **잘한 점** 1-2개 (구체적으로)
2. **놓치기 쉬운 부분** 0-3개 (해당 없으면 생략)
3. **개선 제안** 0-3개 — 실제 적용 시 예상 효과(시간/줄 수)도 추정
4. **학습 일지 업데이트** — 아래 §4 절차 수행

리뷰는 길지 않게. 항목당 1-3줄.

## 4. 학습 일지 업데이트 (`review`/`debug` 후 자동)

`review`나 `debug` 세션이 끝나면 [docs/learning-journal.md](docs/learning-journal.md)에 한 줄 항목을 추가한다. 사용자가 별도 요청하지 않아도 자동 수행.

**기록 규칙:**
- 날짜는 `YYYY-MM-DD` (오늘 날짜는 환경 컨텍스트의 `currentDate` 사용)
- 한 문제당 최대 3줄: 잘한 점 / 놓친 점 / 패턴 태그
- 같은 패턴이 2회 이상 반복되면 상단 **자주 놓치는 패턴** 섹션에 등재 (또는 발생 카운트 +1)

**해소 판정 (중요):**
패턴이 "안 나왔다"고 바로 해소가 아니다. **그 패턴이 발현될 수 있었던 문제**에서 안 나와야 한다.

매 `review`/`debug` 직후, 상단 패턴 표의 각 항목에 대해 판단:
1. **적용 가능했나?** — 이번 문제가 그 패턴을 노출시킬 수 있는 유형인가? (예: "BFS visited 시점" 패턴은 BFS·다중소스 BFS 문제에서만 적용 가능. 순수 DP·문자열 문제에선 적용 불가 → 카운트 변동 없음)
2. **적용 가능했고 안 나왔으면** → "적용 후 무발생" 카운트 +1
3. **적용 가능했고 또 나왔으면** → 발생 카운트 +1, 무발생 카운트 0으로 리셋, 마지막 발생일 갱신
4. **적용 후 무발생 ≥ 3이면 "해소"** — 단순히 3문제가 지나서가 아니라, 노출 기회 3번을 통과했을 때

판정이 애매하면(이 문제에 적용 가능한지 모호) 사용자에게 한 줄로 확인.

**파일 크기 관리:**
- 일별 상세 기록은 최근 ~15문제만 유지. 더 오래된 항목은 패턴 섹션으로 흡수하고 삭제.
- 일지 전체가 200줄 넘으면 즉시 압축.
- 너무 자세히 적지 않는다. 검색·복기 가능한 최소 정보만.

## 5. 개념 노트 자동 기록 ([concepts/](concepts/))

사용자와의 대화에서 **재사용 가능한 지식 조각**이 등장하면 Claude는 `concepts/`에 노트를 자동으로 만들거나 갱신한다. 사용자가 별도 요청하지 않아도 조용히 수행한다.

### 트리거 (YES — 기록한다)

- 사용자가 "왜 이렇게 하나요?", "어떤 게 좋을까요?", "항상 이렇게 하면 되나요?" 식으로 **트레이드오프·관용구·습관**을 물을 때
- 문제 풀이 중 **범용 패턴**이 처음 등장할 때 (방향 벡터, BFS 템플릿, 파싱 관용구, deque 양끝 연산, 회전 공식 등)
- 사용자가 **기본 개념**을 확인하는 질문을 할 때 (예: "1-indexed가 뭐야?", "경계 체크가 뭐야?")
- 사용자가 명시적으로 "이거 기록해줘"라고 할 때

### 비트리거 (NO — 기록 안 한다)

- 한 문제에만 해당하는 세부사항 → 해당 문제의 `notes.md`로
- 교과서·공식 문서에 그대로 있는 뻔한 내용 (예: "`list`와 `tuple`의 차이")
- 단순 확인성 질문 ("이거 맞지?") 중 이미 노트화된 내용
- 특정 버그 디버깅 과정 → `learning-journal.md`로

### 작성 흐름 (매 사용자 턴마다)

1. **먼저 훑는다** — 답변하기 전 또는 답변 직후에 `concepts/` 디렉토리를 확인. 관련 노트가 있으면 그것을 연다.
2. **있으면 갱신** — 기존 노트에 새 예시/함정/트레이드오프를 덧붙인다. 새 파일을 만들지 않는다.
3. **없고 트리거 해당 시 생성** — 아래 템플릿대로 새 파일 작성.
4. **응답 끝에 한 줄로 알린다** — `→ concepts/<파일>.md 갱신` 또는 `→ concepts/<파일>.md 생성`. 노트 내용 전체를 응답에 덧붙이지 않는다.

### 파일 명명

- `concepts/<주제-슬러그>.md` — 영문 소문자 + 하이픈
- 카테고리 번호 prefix를 붙여도 좋다 (기존 `02-bfs-dfs.md` 참고)
- 예: `01-grid-template.md`, `01-direction-vectors.md`, `00-input-parsing.md`

### 노트 템플릿

```markdown
---
trigger: <언제 이 노트를 열어야 하는지 한 줄>
related_problems: [<BOJ번호>, ...]
---

# <제목>

## 언제 쓰나
<한 줄>

## 코드
~~~python
# 5~15줄. 시험장에서 그대로 복붙할 수 있어야 함
~~~

## 함정
- <구체적 실수 + 왜 틀렸는지>

## 관련 개념
- [<다른 노트>](<파일.md>)
```

### 작성 원칙

1. **짧게** — 노트 하나 30줄 이내. 넘으면 분할.
2. **"나만 아는 것"만** — 교과서 지식은 빼고, 사용자가 실제로 헷갈렸거나 실수한 부분만.
3. **예시 중심** — 긴 설명보다 짧은 코드 조각.
4. **함정 섹션 필수** — "이거 빼먹으면 틀린다"가 없으면 안 쓴 것과 같다.
5. **트리거가 분명해야** — 파일 제목·`trigger` 필드만 봐도 "언제 열지"가 명확해야 한다.

### 사용자 오버라이드

- 사용자가 "이번엔 기록하지 마"라고 하면 해당 세션에서만 중단.
- 사용자가 "이 노트 지워"라고 하면 해당 파일 삭제 + 다른 노트의 링크 정리.

### learning-journal과의 역할 분담

| 파일 | 역할 |
|---|---|
| [docs/learning-journal.md](docs/learning-journal.md) | **실수 패턴 추적기.** 반복되는 버그·오해를 카운트하여 약점을 드러낸다. |
| [concepts/](concepts/) | **복붙 가능한 지식 조각.** 시험장에서 꺼내 쓸 관용구·템플릿·트레이드오프. |

두 파일은 목적이 다르니 병행한다. `review`/`debug` 후엔 learning-journal을, 일반 질문 중엔 concepts를 갱신한다. 같은 내용을 양쪽에 중복 기록하지 않는다.

# Python 최적화 체크리스트

A형에서도 Python 시간 보정이 빠듯하다. 매 풀이마다 점검:

- [ ] `input = sys.stdin.readline` (또는 `sys.stdin.read().split()` 일괄 읽기)
- [ ] 재귀 DFS면 `sys.setrecursionlimit(10**6)`
- [ ] BFS는 `collections.deque` (list pop(0) 금지)
- [ ] 우선순위 큐는 `heapq`
- [ ] 출력 많을 땐 `sys.stdout.write` 또는 `'\n'.join(...)` 후 일괄 출력
- [ ] 격자 방문 체크는 `set`보다 `list[list[bool]]`이 빠른 경우 多
- [ ] 함수 호출 오버헤드 큼 — 핫 루프는 인라인 고려

상세는 [templates/io.py](templates/io.py).

# 다른 세션에서의 사용

이 파일(`CLAUDE.md`)은 모든 Claude Code 세션에서 자동으로 컨텍스트로 로드된다.
새 세션에서도 위 워크플로를 그대로 따른다. 사용자가 도움 레벨을 명시하지 않으면 **L1로 가정**하고 시작한다.
